# ⚡ MSPinyin Switcher (基于 ctfmon Hook 的输入法状态控制器)
这是一个基于原生 Hook 和 Windows 输入法底层机制构建的轻量级输入法行为控制器。专为解决中英文输入模式切换不一致、状态丢失等痛点而设计，实现接近零延迟的无感切换。
## 🧠 痛点对齐（本项目解决的问题）
Windows 的输入法切换逻辑在以下要素混合使用时，经常表现出令人抓狂的不一致性：
 * **微软拼音** (基于 TSF / ctfmon 框架)
 * **英语 (美国) - 美式键盘** 布局
 * 在 **浏览器 / IDE / 游戏** 之间进行快速焦点切换
**用户常遇到的高频痛点：**
 * ❌ 使用 Alt + Shift 或 Win + Space 切换布局后，中英状态经常无故脱节。
 * ❌ 微软拼音内部会无规律地随机回退到英文输入状态。
 * ❌ 在 Chromium 系浏览器（如 Chrome/Edge）的 URL 地址栏切换布局时，偶发性导致输入流阻塞/卡死。
 * ❌ 每次切换窗口后，都需要手动去纠正和确认当前的输入法状态。
本项目通过在系统底层**强制执行确定性的输入法状态转换**，彻底消除了上述所有输入摩擦。
## 🚀 核心功能
 * ⚙️ **1. 确定性的输入状态重置**
   在每次系统焦点发生切换时，自动将输入状态重置为 **英语 (美国) - 美式键盘**。从根本上解决“切到新文本框一敲全是乱码中文”或“进游戏按 WASD 弹出输入法”的蠢笨体验。
 * 🇨🇳 **2. 激活输入法时强制默认为中文**
   当手动切回微软拼音时，确保输入法**必定初始为中文输入模式**。同时，完美保留 Shift 键在输入法内部正常切换中英的功能。
 * 🧭 **3. 焦点驱动的全局布局控器**
   利用 Windows 原生事件钩子（SetWinEventHook 监听 EVENT_OBJECT_FOCUS），确保输入法在浏览器、编辑器、大型游戏以及系统原生 UI 之间切换时，状态始终保持绝对的一致。
 * ⚡ **4. 接近零开销的极速设计**
   基于 MinHook 实现直接的函数内联 Hook（Inline Hook），并配合 **AOB 特征码扫描**。热路径（Hot Path）中无任何高配轮询逻辑，运行占用几乎为零。
 * 🧩 **5. 深度的 Windows 输入法底层集成**
   深入拦截 InputService.dll 内部未公开的 IME 服务例程，直接在内存中捕获并接管转换模式状态（Conversion Mode）和当前活动配置（Active Input Profile）的指针，通过底层 Win32 消息链路完美弥补 TSF 框架的设计缺陷。
## 🧬 架构设计概览
```text
ctfmon.exe / InputService.dll
        │
        │  (AOB 扫描特征码 + MinHook 内联拦截)
        ▼
[ 底层 Hook 核心层 ]
        │
        ├── Activate 激活拦截钩子
        ├── Conversion Mode 状态转换钩子
        └── WinEvent 全局焦点监控钩子
        │
        ▼
[ 状态归一化模块 (State Normalizer) ]
        │
        ├── 强制复位 EN-US 美式键盘布局
        ├── 强制锁定 微软拼音初始中文状态
        └── 跨进程同步输入法上下状态机

```
## 🔬 技术亮点（简历加分项）
 * 🪝 **先进的 Hook 拦截策略：**
   采用基于 MinHook 的 **Inline Hook（内联钩子）** 技术进行精准函数拦截。针对 Windows 系统频繁更新、硬编码地址易失效的问题，编写了**动态 AOB 特征码搜索算法**，在运行时动态定位 InputService.dll 内部未公开的核心例程，赋予了项目跨系统版本的强韧性。
 * 🧠 **无开销的运行时状态提取：**
   通过逆向分析微软 InputService 在内存中的数据结构，精准计算出转换模式标志位的**运行时内存偏移量**。通过直接内存指针读写控制状态，**完全绕过了繁重且多层嵌套的 TSF COM 框架开销**，将调度延迟压制在微秒级。
 * 🟟 **系统级上下文感知事件绑定：**
   利用 SetWinEventHook 捕获 EVENT_OBJECT_FOCUS 焦点切换事件，配合 ImmGetDefaultIMEWnd 与 SendMessageTimeoutW 组合拳，实现了**跨进程安全、防挂死**的输入法状态强制调度。
## ⚠️ 兼容性与前置配置
 * **操作系统：** Windows 11 (已在最新的 **24H2** 版本上完美验证，核心特征码两年来历经多次补丁未发生变动)。
 * **输入法环境：**
   * 简体中文 (中国大陆) - 微软拼音
   * 英语 (美国) - 美式键盘
 * **最佳实践建议：**
   系统语言中建议**只保留**上述两种布局，删除其他臃肿项。同时建议将切换语言布局的快捷键从 Ctrl + Shift 改为 Alt + Shift，防止在 FPS 游戏中同时按下蹲（Ctrl）与静步（Shift）时发生按键冲突。
## 📦 快速开始
 1. **编译与获取：** 编译项目生成 mymspinyin.dll，并搭配 cheapInject.exe 注入器使用。
 2. **一键注入：** 运行 cheapInject.exe，程序会瞬间将 DLL 注入到目标系统进程（ctfmon.exe）中，随后注入器自身安全退出，常驻开销完全由宿主承载。
 3. **开机自启：**
   * 在运行窗口（Win + R）输入 shell:startup 回车，打开系统的“启动”文件夹。
   * 将 cheapInject.exe 创建一个快捷方式放入此文件夹，即可实现开机完全无感常驻。

## 🧠 工程设计哲学
> **输入法的状态切换应当是“确定性的”（Deterministic），而不应该是“启发式的”（Heuristic）。**
> 
传统的输入法优化软件总是试图去“猜测用户的意图”，而本项目选择在系统交互的最底层，**直接对输入法的状态机施加决定性的硬性约束**。
## 🧰 技术栈
 * **开发语言：** C++ (Win32 / Native C++)
 * **开源库：** MinHook (Inline Hooking)
 * **核心 Windows API：** SetWinEventHook, ImmGetDefaultIMEWnd, SendMessageTimeoutW, GetKeyboardLayout
 * **逆向工程：** AOB 特征码匹配、运行时内存偏移量解析、Windows 内部输入流机制（ctfmon）探索
## 📌 已知局限
 * 高度依赖 Windows 内部 IME 实现的稳定性，若微软未来对 InputService.dll 进行伤筋动骨的重构，特征码可能需要重新定位。
 * Chrome URL 栏偶发性阻塞的问题，目前采用基于时间戳和状态重置的启发式策略进行缓解。
